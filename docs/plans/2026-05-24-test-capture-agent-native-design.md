## Problem Statement

Developers already manually verify new features before shipping, but that work rarely becomes durable automated coverage. Two weeks later, another change breaks the feature because nobody had time to write the end-to-end test properly.

Test Capture exists to make that manual verification session useful to an AI coding agent. The developer should be able to start Codex, Claude Code, Cursor, or another agent, run `/test-capture`, manually show the feature in a browser, confirm what matters, and have the agent turn that observed behavior into a maintainable test in the repository.

The product is not a replacement for Playwright, Cypress, Vitest, Jest, or a hosted QA platform. It is an agent-native testing workflow that gives the agent the missing sensory layer: what the human actually did, what changed in the UI, which network calls happened, which console errors appeared, which assertions matter, and which steps were only setup.

## What Makes This Cool

The delightful moment is: “I just showed the agent the feature once, and now it understands enough to write the test I was going to skip.”

The tool should feel like a practical extension of the coding agent, not another testing product to manage. It should reuse existing browser and testing infrastructure, keep all artifacts local, and let the agent combine capture evidence with repo context, Jira/Linear/GitHub MCPs, hidden markdown docs, and existing test conventions.

The unusual part is the human-agent handoff. Most recorders capture clicks. Test Capture captures a developer’s verification intent and exposes it to an agent that can actually edit the repo.

## Constraints

- V1 must be one complete usable product, not a staged demo.
- The primary workflow starts inside an AI coding agent.
- The agent owns external context gathering: tickets, docs, repo inspection, and existing tests.
- Test Capture owns browser-session evidence and human intent markers.
- Playwright should be the first capture engine because it provides browser control, traces, screenshots, network capture, accessibility snapshots, and a mature test ecosystem.
- Everything must run locally by default.
- Privacy and redaction are part of the core product, not a later add-on.
- The tool should work through an agent-native interface when available and through CLI/files when not.

## Premises

1. The product is an agent-native testing workflow, not a recorder.
2. Capture is useful because Codex, Claude, Cursor, or another coding agent can use it to create and run real tests.
3. The agent should fetch Jira, Linear, GitHub Issue, markdown, and repo context through its normal tools or MCPs.
4. Test Capture should provide the missing sensory layer: browser evidence from a human’s manual verification session.
5. V1 must complete the loop: capture, confirm, plan coverage, hand off, write test, run test, triage failures, report result.
6. The first audience is developers already using AI coding agents and already feeling the cost of skipped E2E tests.

## Approaches Considered

### Approach A: Agent Context CLI

Build a CLI that launches Playwright, records the manual session, writes a capture bundle, and produces `agent-context.md` for Codex or Claude to consume.

Pros:
- Fastest path to a real open-source release.
- Works even where MCP support is limited.
- Simple to debug because everything is local files.

Cons:
- The agent handoff can feel manual.
- Querying a large session is clumsy.
- The product risks feeling like a context generator rather than an agent tool.

### Approach B: Agent-Native Tool Suite

Build Test Capture as both a CLI and an agent-facing tool interface. The agent can start and stop sessions, query steps, inspect screenshots, read network and console evidence, and drive the test-writing loop.

Pros:
- Best match for the actual product vision.
- Makes Test Capture feel like part of Codex, Claude, Cursor, or similar agents.
- Lets the agent pull only the evidence it needs instead of loading huge traces into context.
- Gives contributors clear extension points.

Cons:
- More moving parts in V1.
- Requires careful API boundaries before real user feedback.
- Agent tool support differs across environments, so the fallback path must be first-class.

### Approach C: Skill/Workflow First

Start with a polished Codex/Claude workflow that orchestrates existing Playwright tools, repo inspection, intent confirmation, and test creation.

Pros:
- Fast to dogfood.
- Easy to demonstrate.
- Closest to the user’s daily workflow.

Cons:
- Less portable across agents.
- May look like a prompt workflow instead of an open-source tool.
- Later extraction into a reusable engine could create rework.

## Recommended Approach

Use Approach B: Agent-Native Tool Suite, with Approach A’s file-based fallback built in from day one.

The V1 product should have two equal surfaces:

1. Agent tools for the ideal workflow.
2. CLI and local artifacts for portability, debugging, and unsupported agents.

The agent-native surface should expose session operations:

- `start_capture`
- `stop_capture`
- `list_sessions`
- `get_session_summary`
- `get_step`
- `get_screenshot`
- `get_network_events`
- `get_console_events`
- `get_selector_candidates`
- `get_coverage_plan`
- `get_testability_findings`
- `triage_test_failure`
- `get_ledger`
- `write_agent_context`

The CLI should mirror the same capabilities:

- `test-capture start --url http://localhost:3000`
- `test-capture stop`
- `test-capture summary <session>`
- `test-capture step <session> <n>`
- `test-capture network <session>`
- `test-capture screenshots <session>`
- `test-capture selectors <session>`
- `test-capture coverage-plan <session>`
- `test-capture testability <session>`
- `test-capture triage <session> --test-output <file>`
- `test-capture ledger`
- `test-capture doctor`
- `test-capture clean`

The internal architecture should keep these surfaces thin over the same core modules:

- `recorder`: Playwright browser launch, actions, screenshots, trace, network, console.
- `session-store`: local artifact management under `.test-capture/sessions`.
- `intent`: human markers, scenario summary, assertion candidates, uncertainty detection.
- `repo`: test framework detection, package scripts, Playwright config, existing test conventions.
- `coverage-plan`: proposed tests, assertions, setup needs, flake risks, and explicit non-goals for the agent to approve before code generation.
- `testability`: advisory findings about accessible names, labels, selector stability, timing risks, and app changes that would make the feature easier to test.
- `failure-triage`: generated-test failure classification using capture evidence, test output, console logs, and network events.
- `ledger`: local mapping from capture sessions to generated tests and latest verification status.
- `agent`: MCP/tool adapter, CLI output, `agent-context.md` generation.
- `privacy`: redaction, sensitive-field masking, `.test-captureignore`.

Artifact provenance should be explicit:

- `tool_generated`: deterministic facts produced by Test Capture
- `agent_authored`: summaries, coverage plans, and triage narratives drafted by the coding agent
- `human_approved`: developer-confirmed scenarios, coverage decisions, and final linked tests

Every persisted artifact should declare its provenance and source inputs.

## V1 Feature Set

### Agent Entry Point

- Intended UX: `/test-capture` from an agent session.
- Agent adapter through MCP or equivalent tool interface.
- CLI fallback for all core operations.
- No built-in Jira/Linear/GitHub integration. The agent uses existing MCPs or tools for that.

### Session Setup

- Accept target URL.
- Accept optional human description or ticket reference.
- Detect repo root, branch, changed files, package manager, Playwright config, test folders, and likely test commands.
- Detect whether Playwright is already installed and provide actionable setup output if not.

### Manual Browser Capture

- Launch Chromium with Playwright.
- Let the developer manually interact with the app.
- Capture clicks, typing, navigation, form values, selected elements, screenshots, accessibility snapshots, console logs, network events, response status codes, redirects, timing, and trace data.
- Support auth-friendly local testing through persistent browser context or existing storage state.

### Human Intent Markers

During capture, the developer can mark:

- assert this
- ignore this step
- this was setup
- this is the bug
- this should persist after reload
- split this into another test

Markers may be added through a small browser overlay, terminal hotkeys, or agent commands.

V1 should prefer terminal hotkeys and agent commands for markers. Browser overlay support should be optional because overlays can perturb DOM, screenshots, selectors, and accessibility snapshots.

### Stop And Summarize

- Generate a human-readable scenario.
- Identify setup, preconditions, user role, main behavior, expected result, suggested assertions, and flake risks.
- Ask the developer to confirm or edit the scenario.
- Flag uncertainty rather than inventing business rules.

### Coverage Plan Gate

Before the agent writes code, Test Capture should propose a coverage plan for developer approval.

The coverage plan should include:

- test file or files to create
- whether the scenario should become one test or multiple tests
- exact assertions to include
- setup data and fixtures likely needed
- helpers or repo conventions the agent should inspect
- what not to test
- known flake risks
- open questions that should block generation

The agent should only move to test creation after the developer approves or edits this plan.

### Testability Inspector

Every capture should include advisory testability findings.

Examples:

- missing accessible names
- unlabeled inputs
- brittle selector candidates
- generated CSS chains
- table rows without stable identity
- transient UI that is unsafe to assert directly
- missing fixture or setup helper signals

These findings should be advisory. The coding agent may suggest app changes, but it must not silently edit app code unless the developer approves.

### Agent Context Bundle

Write:

```txt
.test-capture/sessions/<session-id>/
  session.json
  agent-safe-index.json
  scenario.md
  coverage-plan.md
  testability.md
  agent-context.md
  network.har
  console.log
  screenshots/
  trace.zip
```

`agent-context.md` is the main handoff artifact. It should tell the agent what happened, what matters, what to avoid overfitting to, where evidence lives, and what questions remain.

`agent-safe-index.json` is the query layer for coding agents. It should contain redacted, structured references to actions, selector candidates, network summaries, console summaries, screenshots, timing, human markers, state transitions, and artifact paths.

The index should have:

- schema version
- stable event ids
- stable artifact ids
- redaction summary
- size limits for large fields
- provenance for each generated field
- fixture tests that lock the schema

Data flow:

```txt
Browser Event
  │
  ▼
Raw Capture Buffer
  │
  ├─ if SECRET detected ──▶ drop / mask / fail closed
  │
  ▼
Redaction Pipeline
  │
  ▼
Artifact Store
  │
  ├─ redacted local artifacts
  │
  └─ agent-safe index
          │
          ▼
    MCP/CLI query tools
          │
          ▼
    Coding Agent
```

Shadow paths:

- nil event: ignore and log debug output
- empty session: refuse summary and keep the session out of coverage planning
- redaction error: fail closed and do not persist raw artifact
- artifact missing: raise `SessionArtifactNotFoundError`
- agent query requests sensitive artifact: return redacted summary unless an explicit local path request is approved

### Queryable Agent Interface

The agent should not need to load a huge trace or HAR into context. It should query only the relevant parts of the session.

Required query operations:

- list sessions
- get summary
- inspect step
- fetch screenshot path
- inspect network calls
- inspect console output
- inspect selector candidates
- inspect uncertainty and open questions
- inspect coverage plan
- inspect testability findings
- inspect ledger entries

### Repo-Aware Test Handoff

- Collect lightweight repo signals: Playwright config, test folders, package scripts, and nearby existing tests.
- Surface likely helpers, fixtures, auth setup, naming conventions, and preferred selector style.
- Tell the coding agent to verify these repo signals before editing tests.
- Suggest test file location.
- Instruct the agent to use existing repo style.
- Instruct the agent not to convert raw clicks into brittle code.

### Agent Test Creation Loop

V1 should support this full loop:

1. Capture session.
2. Confirm scenario.
3. Review and approve the coverage plan.
4. Agent writes the test.
5. Agent runs the relevant test command.
6. Test Capture helps classify failures using capture evidence.
7. Agent fixes failures caused by the new test.
8. Agent reports final test file, command run, pass/fail state, and remaining risks.

The tool should not ship its own LLM in V1. Interpretation is agent-mediated:

- Test Capture records, indexes, redacts, and exposes evidence.
- Test Capture produces deterministic hints: selector candidates, network events, console events, timing, human markers, redaction status, and artifact paths.
- The active coding agent drafts the scenario summary, coverage plan, and failure triage narrative from those facts.
- Test Capture stores the final approved artifacts.

### Failure Triage Loop

When the generated test fails, Test Capture should help the agent classify the likely failure type:

- generated test is wrong
- app behavior differs from captured behavior
- auth or setup data is missing
- selector is brittle
- timing or animation flake
- local app/server is unavailable
- capture intent was misunderstood
- expected behavior needs human clarification

The triage output should reference concrete evidence: failing assertion, captured screenshot, nearby network events, console events, and relevant intent markers.

### Local Coverage Ledger

Maintain a minimal local ledger that maps capture sessions to generated tests and latest verification status.

Example:

```json
[
  {
    "sessionId": "customer-billing-email-2026-05-24",
    "scenario": "Admin edits customer billing email",
    "generatedTests": ["e2e/customer-billing-email.spec.ts"],
    "status": "passing",
    "lastVerified": "2026-05-24T10:30:00Z"
  }
]
```

This is local metadata only, not a dashboard or test management system.

### Session State Machine

Every session should move through explicit states:

```txt
CREATED
  │
  ▼
RECORDING
  │
  ├─ stop_capture
  ▼
CAPTURED
  │
  ├─ agent drafts scenario
  ▼
SCENARIO_DRAFTED
  │
  ├─ developer approves/edits
  ▼
SCENARIO_APPROVED
  │
  ├─ agent drafts coverage plan
  ▼
COVERAGE_PLANNED
  │
  ├─ developer approves/edits
  ▼
COVERAGE_APPROVED
  │
  ├─ agent writes test
  ▼
TEST_GENERATED
  │
  ├─ test run passes
  ▼
VERIFIED
  │
  └─ test run fails
      ▼
TRIAGE_NEEDED
        │
        ├─ agent fixes test
        └─ developer clarification needed
```

Invalid transitions should fail loudly:

- cannot generate a test before coverage approval
- cannot mark verified without a linked test file and passing command
- cannot triage failure without test output
- cannot edit a finalized session without creating a new revision

### Test Quality Guardrails

- Prefer role, label, text, and test-id selectors.
- Warn on generated CSS chains.
- Distinguish setup steps from behavior under test.
- Suggest assertions based on confirmed intent, network success, UI state, persistence, and validation errors.
- Detect likely flake sources: animation, arbitrary timeouts, dynamic IDs, random data, shared state, and missing cleanup.

### Privacy And Redaction

- Local-only by default.
- No cloud storage.
- Redact cookies, auth headers, tokens, passwords, secrets, and sensitive storage values.
- Mask sensitive input fields.
- Support `.test-captureignore`.
- Warn that screenshots and HAR files may contain sensitive data.
- Recommend adding `.test-capture/` to `.gitignore`.
- Warn if `.test-capture/` is not ignored by git.

Data classification:

```txt
SAFE
- action type
- URL path without query secrets
- HTTP method
- status code
- accessible role/name after redaction
- selector candidates after redaction

SENSITIVE
- screenshots
- HAR
- request/response payload metadata
- console logs
- local/session storage keys
- typed values

SECRET
- cookies
- Authorization headers
- Set-Cookie
- tokens
- passwords
- API keys
- full storage values
- request/response bodies unless explicitly enabled
```

Default behavior:

- never persist `SECRET`
- persist `SENSITIVE` only after redaction
- default to `--privacy strict`
- require explicit per-session consent before capturing screenshots, HAR bodies, storage values, and typed text values
- require explicit opt-in before persisting full request or response bodies

### Error And Rescue Registry

V1 should define named errors and fail closed when privacy or artifact integrity is at risk.

```txt
CODEPATH                    | WHAT CAN GO WRONG                  | ERROR NAME
----------------------------|------------------------------------|----------------------------
start_capture               | target URL missing/invalid         | InvalidTargetUrlError
start_capture               | browser launch fails               | BrowserLaunchError
start_capture               | app unreachable                    | TargetUnreachableError
record_event                | browser disconnects                | BrowserDisconnectedError
write_session_artifact      | filesystem write fails             | ArtifactWriteError
redact_capture_data         | redaction cannot complete safely   | RedactionFailedError
get_step / get_screenshot   | session or artifact missing        | SessionArtifactNotFoundError
approve_coverage_plan       | invalid state transition           | InvalidSessionTransitionError
triage_test_failure         | test output missing/unparseable    | TestOutputParseError
update_ledger               | linked test file missing           | LedgerConsistencyError
```

Rules:

- redaction failures must stop artifact persistence unless the developer explicitly retries with safer capture settings
- artifact write failures must prevent the session from advancing to `CAPTURED`
- invalid state transitions must return a named error and preserve the current state
- ledger consistency failures must not mark a session as verified
- every error shown to the user should include the failed operation, the session id if available, and the next safe action

### Developer UX

- `test-capture doctor` checks browser, Playwright, repo, config, and agent adapter readiness.
- `test-capture clean` removes old sessions.
- Config file for defaults.
- Clear final report after every capture.
- Works even if the final agent handoff is manual.
- Store session data in a storyboard-friendly shape so a future replay viewer can be built without redesigning the artifact format.
- Provide both human-readable and machine-readable doctor output, e.g. `test-capture doctor --json`.
- Every failed doctor check should include the failed check, likely cause, and exact next command or doc link.

Playwright dependency strategy:

- Test Capture should use its own Playwright dependency for capture runtime stability.
- Generated tests should use the target repo’s Playwright dependency and conventions.
- The handoff should warn when the capture Playwright version and repo Playwright version differ materially.

Session storage reliability:

- write artifacts atomically
- use a session lock while recording or updating state
- treat finalized sessions as immutable
- create a revision when a human edits an approved scenario or coverage plan
- define interrupted capture recovery behavior

`test-capture doctor` should check:

- supported Node version
- detected package manager
- Playwright availability for the capture engine
- browser install availability
- git repository status
- whether `.test-capture/` is ignored by git
- MCP server startup
- CLI write access to the session directory
- target URL reachability when provided
- existing Playwright tests
- likely test command

Final capture report should include:

- session id
- current state
- artifact directory
- privacy mode
- redaction count
- secret persistence failures
- action count
- screenshot count
- console warning/error count
- network event count
- testability finding count
- coverage plan status
- generated test paths
- last test command and pass/fail result

## Scope Expansion Decisions

Accepted into V1:

- Coverage Plan Gate before test generation.
- Advisory Testability Inspector for every capture.
- Failure Triage Loop for generated-test failures.
- Minimal local Coverage Ledger mapping capture sessions to generated tests and latest status.

Deferred with hooks:

- Replay / Storyboard Viewer. V1 should structure session data so this can be added later, but should not include a polished viewer.

Not in scope:

- SaaS dashboard.
- Built-in Jira, Linear, or GitHub issue integrations.
- Cypress-first implementation.
- Self-healing tests.
- Fully autonomous app-code edits based on testability findings.

## Open Questions

- Should the human intent UI be a browser overlay, terminal hotkeys, or both?
- Should the capture engine use its own Playwright dependency while generated tests use the repo’s Playwright dependency?
- Should screenshots be enabled by default with strong warnings, or opt-in for stricter privacy?
- How much repo inspection should Test Capture perform before it starts duplicating work the coding agent can already do?

## Success Criteria

V1 is successful if a developer can:

1. Start from an AI coding agent session.
2. Run the Test Capture workflow against a local app.
3. Manually verify a feature in the browser.
4. Mark what matters during the session.
5. Confirm the generated scenario.
6. Review and approve a coverage plan.
7. Have the agent write a maintainable Playwright test in the repo’s style.
8. Run the relevant test command.
9. Triage and fix generated-test failures using capture evidence.
10. End with a test file the developer would keep.

The first validation target should be 10 real feature flows across 3 to 5 repos. At least 7 of those should produce tests that need only minor human edits.

Implementation acceptance criteria:

- `start_capture` and `stop_capture` produce a complete session bundle.
- no `SECRET` data is persisted in default mode.
- `agent-safe-index.json` validates against the versioned schema.
- MCP and CLI can both query the same session.
- coverage generation is blocked until coverage approval exists.
- ledger cannot mark verified without linked test path and passing command.
- `doctor --json` returns machine-readable check results.
- a fixture app dogfood test exercises capture, query, coverage gate, failure triage, and ledger update.
